# Claude Code 开发团队角色协作系统

通过 SubAgent（子进程）实现真正隔离的团队角色协作。每个角色运行在独立上下文中，互不干扰。

## 快速开始

```bash
# 复制 .claude/ 目录到你的项目
cp -r .claude/ /path/to/your/project/
```

## 架构

```
用户 ←→ 主对话（轻量调度器）
              │
              ├→ Agent 子进程: 项目经理   (独立 200k 上下文)
              ├→ Agent 子进程: 程序员     (独立 200k 上下文)
              ├→ Agent 子进程: 代码审查员 (独立 200k 上下文)
              └→ Agent 子进程: 测试员     (独立 200k 上下文，可选)
              │
              └── 角色间通过 .claude/tasks/ 文件系统通信
```

## 角色

| 命令          | 角色                  | 职责                                                       |
| ------------- | --------------------- | ---------------------------------------------------------- |
| `/项目经理`   | 首席架构师 & 技术总监 | 需求分析、阅读实际代码、架构设计、任务拆解                 |
| `/程序员`     | 顶级全栈开发专家      | 代码实现、可质疑设计缺陷、自检后交付                       |
| `/代码审查员` | 首席质量架构师        | 独立上下文代码审查、安全审计                               |
| `/测试员`     | 首席测试工程师        | 测试编写与执行（**可选**，需 conventions.md 定义测试策略） |

## 核心机制

- **项目经理定方向，程序员定实现** — 项目经理负责"做什么"和关键约束，程序员全权决定"怎么做"
- **项目经理必须读代码** — 不能只看文档就做设计，必须阅读实际代码理解现有实现
- **项目经理评估拆分** — 预估改动超过 10-15 个文件时，必须拆分为多个独立任务，各任务独立开发→审查→测试
- **项目经理设计自省** — 设计完成后必须回答盲区检查问题，有"不确定"必须写入待确认事项
- **程序员可质疑设计** — 发现设计遗漏时在 dev-plan.md 中反馈，不盲目照搬
- **程序员必须自检** — 11 项自检清单（含回归检查、变更摘要更新），写完代码先过一遍再交付
- **程序员有质量红线** — 不重复、不硬编码、不忽略错误、不过度设计、不留 TODO
- **程序员勾选任务完成** — 每阶段完成后在 dev-tasks.md 中勾选 checkbox，代码审查员据此验证
- **代码审查员独立审查** — 隔离上下文，零偏见，通过代码变更摘要精确定位审查范围
- **测试员可选参与** — 项目是否需要测试由 conventions.md 中的测试策略决定
- **测试员只写测试不改业务代码** — 发现 bug 记录到 test-report.md，由程序员修复
- **设计反馈闭环** — 程序员/代码审查员/测试员反馈设计问题 → 用户决定是否呼叫项目经理修改

## 协作流程

```
用户需求 (开发分支)
    │
    ▼
/项目经理 → design.md + dev-tasks.md + status.md
    │
    ▼
/程序员 → dev-plan.md + 代码  ◄──── 审查回路 ────┐
    │                                         │
    ▼                                         │
/代码审查员 → review-log.md                     │
    │                                         │
    ├─ 有问题 → /程序员修复 ─────────────────────┘
    │
    └─ 通过
        │
        ▼
/测试员（可选）→ test-report.md  ◄── 测试回路 ────┐
    │                                         │
    ├─ 有失败 → /程序员修复（不经过代码审查员）─────┘
    │
    └─ 全部通过
        │
        ▼
用户合并到 main/master → /项目经理 更新 docs/
```

## 文件通信

角色间通过 `.claude/tasks/` 目录下的文件进行通信：

```
.claude/tasks/{YYYYMMDD_功能名}/
├── design.md       # 项目经理 → 程序员/测试员：架构设计 + 关键约束 + 测试策略建议
├── dev-tasks.md    # 项目经理定义任务，程序员勾选完成：分阶段任务清单 + 验收标准
├── status.md       # 项目经理创建，程序员/测试员更新：全局状态 + 变更记录
├── dev-plan.md     # 程序员 → 代码审查员/测试员：开发计划 + 变更摘要 + 设计反馈
├── review-log.md   # 代码审查员 → 程序员：审查问题 + 设计反馈评估
└── test-report.md  # 测试员 → 程序员：测试报告 + 失败项详情（可选）
```

## 流程示例

<details>
<summary><b>示例 A：新功能完整流程</b>（设计 → 开发 → 审查 → 测试 → 需求变更 → 收尾，14 步）</summary>

以"用户登录功能"为例，展示包含审查问题修复、测试和需求变更的完整流程。

**第一轮：设计 → 开发 → 审查**

```
① /项目经理 实现用户登录功能（邮箱+密码，JWT鉴权）

   调度器先做需求澄清（如信息不足会追问），然后启动子进程。
   子进程读取 docs/ 全量文档 + 涉及模块代码 → 产出：
   - tasks/20260312_用户登录/design.md      ← 架构设计 + 影响分析 + 测试策略建议 + 设计自省
   - tasks/20260312_用户登录/dev-tasks.md   ← 分阶段任务清单
   - tasks/20260312_用户登录/status.md      ← "设计完成，等待开发"
   调度器展示摘要，提示 → /程序员

② /程序员 .claude/tasks/20260312_用户登录/

   子进程读取 design.md + dev-tasks.md + docs/ 全量文档 + 现有代码
   → 创建 dev-plan.md（开发计划）
   → 更新 status.md → "开发中"
   → 按阶段实现代码，每阶段完成后执行 11 项自检
   → 在 dev-tasks.md 中勾选已完成的任务
   → 更新 dev-plan.md 中的"代码变更摘要"表格
   → 输出摘要，提示 → /代码审查员

③ /代码审查员 .claude/tasks/20260312_用户登录/

   子进程在隔离上下文中：
   → 读 design.md（理解预期行为）
   → 读 dev-tasks.md（检查 checkbox 完成状态 + 验收标准）
   → 读 dev-plan.md 变更摘要（定位审查范围）
   → 读 conventions.md + 按需读其他 docs/（验证架构一致性）
   → 逐文件深度审查（功能完整性、代码质量、架构一致性、安全性、性能、现有功能影响）
   → 产出 review-log.md
```

**第二轮：审查发现问题 → 修复 → 复审**

```
   审查员发现：🔴 密码未加盐哈希、🟡 JWT 过期时间硬编码、🟢 变量命名不一致
   输出 "需修改"，提示 → /程序员

④ /程序员 .claude/tasks/20260312_用户登录/

   子进程读取 review-log.md → 逐项修复 3 个问题
   修复过程中发现：design.md 未定义 refresh token 机制
   → 修复代码 + 执行自检 + 更新变更摘要
   → 在 dev-plan.md "设计问题反馈"中记录 refresh token 遗漏
   → 输出摘要（醒目标注设计问题），提示 → /代码审查员

⑤ /代码审查员 .claude/tasks/20260312_用户登录/   （复审）

   → 区分审查轮次（轮次1 已修复 vs 轮次2 复审）
   → 验证 3 个问题均已修复 ✅
   → 评估程序员的设计反馈："refresh token 遗漏确实合理，建议项目经理评估"
   → 输出 "通过（有设计反馈待处理）"
   → 项目有测试策略，提示 → /测试员
```

**第三轮：测试（可选，项目有测试策略时）**

```
⑥ /测试员 .claude/tasks/20260312_用户登录/

   → 读 conventions.md（获取测试框架和覆盖率要求）+ 按需读其他 docs/
   → 读 design.md（理解功能预期行为和边界条件）
   → 读 dev-plan.md 变更摘要（定位要测试的代码）
   → 制定测试策略：单元测试覆盖认证逻辑，API 测试覆盖登录端点
   → 编写测试代码 → 执行测试
   → 发现：🔴 密码重置接口未校验 token 过期、🟡 并发登录未限制
   → 产出 test-report.md，提示 → /程序员

⑦ /程序员 .claude/tasks/20260312_用户登录/

   子进程读取 test-report.md → 逐项修复 2 个问题
   → 修复代码 + 执行自检 + 更新变更摘要
   → 输出摘要，提示 → /测试员（不经过代码审查员）

⑧ /测试员 .claude/tasks/20260312_用户登录/   （复测）

   → 重新执行之前失败的测试 → 全部通过 ✅
   → 评估是否需要补充新测试用例
   → 输出 "全部通过"
```

**第四轮：需求变更 → 重新开发 → 审查 → 测试**

```
   用户决定采纳反馈，并追加新需求："加上 OAuth 第三方登录"

⑨ /项目经理 在 20260312_用户登录 任务中追加 OAuth + refresh token

   子进程走"场景三：需求变更"：
   → 读取 docs/ 全量文档 + 任务目录现有文档 + 涉及模块代码，重新理解项目现状
   → 更新 design.md（标注新增章节：OAuth 流程、refresh token 机制）
   → 更新 dev-tasks.md（新增任务用 [ ]，已完成任务的 [x] 保留不动）
   → 更新 status.md 变更记录（标注 dev-plan.md/test-report.md 中哪些内容因此过时）
   → 输出摘要（醒目提示变更影响），提示 → /程序员

⑩ /程序员 .claude/tasks/20260312_用户登录/

   → 读取 docs/ 全量文档 + 更新后的 design.md 和 status.md 变更记录
   → 清理 dev-plan.md 中过时内容，基于新设计重新规划
   → 实现 OAuth + refresh token → 自检 → 更新变更摘要
   → 提示 → /代码审查员

⑪ /代码审查员 → 通过 ✅ → /测试员
⑫ /测试员 → 补充 OAuth 测试 → 全部通过 ✅
```

**收尾：合并 → 更新文档**

```
⑬ 用户手动合并开发分支到 main/master

⑭ /项目经理 更新项目文档

   调度器检查分支 → 确认在 main/master 上
   → 子进程读取 docs/ 全量文档 + 任务的 design.md 和 status.md
   → 更新 docs/changelog.md、相关文档、status.md
```

<summary><b>示例 B：全新项目首次开发</b>（初始化 → 设计 → 开发 → 审查，无测试策略，4 步）</summary>

在一个刚 `npm init` 的空项目上，从零开始。

```
① /项目经理 实现一个 REST API，支持用户 CRUD   （在 main 分支上执行）

   → 用户同时给出了功能需求，继续走"场景二"：
   → 阅读项目现有代码（此时几乎为空）
   → 创建 tasks/20260313_用户CRUD/design.md
   → 创建 tasks/20260313_用户CRUD/dev-tasks.md
   → 创建 tasks/20260313_用户CRUD/status.md
   → 输出摘要，提示 → /程序员

② /程序员 .claude/tasks/20260313_用户CRUD/

   → 读取任务文档 + docs/ 全量文档（刚初始化的 2 个文件）
   → 创建 dev-plan.md → 实现代码 → 自检 → 勾选 checkbox → 更新变更摘要
   → 无测试策略 → 跳过"运行测试"自检项
   → 输出摘要，提示 → /代码审查员

③ /代码审查员 .claude/tasks/20260313_用户CRUD/

   → 读任务文档 + conventions.md
   → 逐文件审查 → 全部通过 ✅
   → 无测试策略 → 提示"可以合并代码，之后输入 /项目经理 更新项目文档"

④ 用户合并到 main → /项目经理 更新项目文档

   → 读取 docs/ 全量 + 任务文档 → 更新 changelog.md、status.md
```

**关键点**：场景一需在 main 分支执行（要创建 docs/）。场景一自动衔接场景二，一次调用完成项目初始化 + 任务设计。无测试策略时跳过测试员环节。

<summary><b>示例 C：已有复杂项目重构</b>（大范围改动 + 多轮审查 + 设计反馈链，8 步）</summary>

在一个已有 10+ docs/ 文档的电商项目中，将订单支付从同步改为异步。

```
① /项目经理 重构订单模块的支付流程，从同步改为异步

   → 读取 docs/ 全量文档（index.md, conventions.md, architecture.md,
     api-guide.md, data-model.md, payment-flow.md 等 10+ 文件）
   → 阅读支付模块、订单模块、通知模块的实际代码
   → 评估任务规模：涉及 5 个模块、12 个文件，在 10-15 阈值边界
     → 判断：虽然文件数接近阈值，但都围绕同一支付流程，耦合紧密，拆分反而增加协调成本
     → 决定不拆分，作为单个任务处理
   → 创建 tasks/20260313_支付异步重构/design.md
     - "现有代码分析"章节：详细记录当前同步支付的调用链
     - "影响分析"章节：涉及 5 个模块、12 个文件
     - "设计自省"：标注可能遗漏的 webhook 回调模块
   → 产出 dev-tasks.md（3 个阶段）、status.md
   → 提示 → /程序员

② /程序员 .claude/tasks/20260313_支付异步重构/

   → 读取 docs/ 全量文档 + 任务文档（全面理解项目架构后再动手）
   → 阅读涉及的 12 个文件的现有代码
   → 创建 dev-plan.md → 按 3 个阶段实现
   → 每阶段：实现 → 自检（特别关注第 9 条"对现有功能无破坏性影响"）→ 勾选 → 更新变更摘要
   → 有测试策略 → 运行现有测试确保不破坏旧功能
   → 提示 → /代码审查员

③ /代码审查员 .claude/tasks/20260313_支付异步重构/

   → 读 dev-plan.md 变更摘要（12 个文件）→ 逐文件深度审查
   → 发现：🔴 异步回调的幂等性未处理、🟡 旧同步接口未做兼容层
   → 输出 "需修改"，提示 → /程序员

④ /程序员 修复审查问题

   → 读取 review-log.md → 修复 2 个问题
   → 修复完成后执行自检 + 更新变更摘要
   → 提示 → /代码审查员

⑤ /代码审查员 复审 → 通过 ✅ → 有测试策略 → /测试员

⑥ /测试员 .claude/tasks/20260313_支付异步重构/

   → 读 conventions.md + design.md 影响分析
   → 制定测试策略：
     - 单元测试：异步消息序列化/反序列化、幂等性检查
     - 集成测试：支付回调完整流程
     - E2E：项目未配 headless 浏览器 → 记录到"建议手动验证"
   → 编写 + 执行测试 → 全部通过 ✅

⑦ 用户合并到 main

⑧ /项目经理 更新项目文档

   → 读取 docs/ 全量文档 → 更新 architecture.md（支付流程变更）、
     payment-flow.md（全面重写）、changelog.md、status.md
```

**关键点**：项目经理会评估是否需要拆分任务（本例 12 文件在阈值边界，因耦合紧密决定不拆）。docs/ 全量读取在重构场景中至关重要 — 项目经理和程序员需要理解 architecture.md、data-model.md 等才能安全重构。修复审查问题后也执行自检，防止引入新问题。E2E 测试因环境限制记录为手动验证项。

<summary><b>示例 D：大规模任务拆分</b>（PM 拆分 → 多任务独立循环，6 步）</summary>

将整个项目从 REST 改为 GraphQL，涉及 25+ 文件，PM 评估后拆分为多个独立任务。

```
① /项目经理 将所有 API 端点从 REST 迁移到 GraphQL

   → 读取 docs/ 全量文档 + 所有 API 相关代码
   → 评估任务规模：涉及 6 个模块、25+ 个文件 → 远超 10-15 阈值 → 必须拆分
   → 按模块独立性拆分为 3 个子任务：
     - tasks/20260313_GraphQL_用户模块/（无依赖，可先开发）
     - tasks/20260313_GraphQL_订单模块/（依赖用户模块的 GraphQL schema）
     - tasks/20260313_GraphQL_支付模块/（依赖订单模块）
   → 每个子任务目录各自创建 design.md、dev-tasks.md、status.md
   → 各 design.md 中标注子任务间依赖关系和接口契约
   → 输出摘要：3 个子任务 + 依赖顺序 + 建议开发顺序

② /程序员 .claude/tasks/20260313_GraphQL_用户模块/

   → 按正常流程开发第一个子任务（无外部依赖）
   → 自检 → 勾选 → 更新变更摘要

③ /代码审查员 .claude/tasks/20260313_GraphQL_用户模块/

   → 审查通过 ✅

④ /程序员 .claude/tasks/20260313_GraphQL_订单模块/

   → 读取用户模块的 design.md 了解接口契约
   → 开发第二个子任务 → 自检 → 审查...

⑤ 依次完成所有子任务的 开发 → 审查 → 测试 循环

⑥ 全部完成后合并到 main → /项目经理 更新项目文档
```

**关键点**：大规模改动（25+ 文件）被拆为 3 个独立任务，每个任务独立走完整的 开发→审查→测试 流程。子任务间通过 design.md 中的接口契约解耦，按依赖顺序开发。

### 文档一致性保障

| 场景                 | 谁负责                      | 怎么保障                                                                               |
| -------------------- | --------------------------- | -------------------------------------------------------------------------------------- |
| 开发/审查/测试的进度 | 程序员/测试员更新 status.md | 程序员和测试员各有 status.md 写权限（仅各自的进度部分）                                |
| 需求变更后旧文档过时 | 项目经理标注 + 程序员清理   | 项目经理在 status.md 标注哪些内容过时（含 test-report.md），程序员据此清理 dev-plan.md |
| 多轮审查记录混淆     | 代码审查员区分轮次          | 通过日期和阶段标识区分，读 status.md 了解变更历史                                      |
| 多轮测试记录混淆     | 测试员区分轮次              | 通过日期和阶段标识区分，读 status.md 了解变更历史                                      |
| 修复阶段发现设计问题 | 程序员/测试员记录           | 程序员写入 dev-plan.md，测试员写入 test-report.md，用户决定是否呼叫项目经理            |

## 目录结构

```
.claude/
├── CLAUDE.md              # 主文档（架构说明、规则、流程）
├── settings.local.json    # 本地权限配置
├── commands/              # 角色命令（SubAgent 调度器）
│   ├── 项目经理.md
│   ├── 程序员.md
│   ├── 代码审查员.md
│   └── 测试员.md
├── templates/             # 文档模板（从 Prompt 中抽离，按需读取）
│   ├── design-template.md
│   ├── dev-tasks-template.md
│   ├── dev-plan-template.md
│   ├── review-log-template.md
│   ├── test-report-template.md
│   └── status-template.md
├── docs/                  # 项目文档（由项目经理维护，仅主分支更新）
└── tasks/                 # 任务文档目录（任何分支可操作）
```

## 注意事项

- **子进程运行中不能追问** — 项目经理调度器会在启动前先做一轮需求澄清；子进程遇到不确定的问题会写入"待确认事项"，看完后可以再调一次补充
- **过程不可见** — 只能看到最终摘要，但所有产出都在 `.claude/tasks/` 文件中，随时可读
- **沟通语言** — 所有角色与用户对话使用中文
- **分支规则** — `tasks/` 目录任何分支可操作，`docs/` 目录仅在 main/master 分支更新
- **测试员可选** — 只有在 `conventions.md` 中定义了测试策略的项目才需要使用测试员
