# 开发团队角色协作系统

通过 SubAgent（子进程）实现真正隔离的团队角色协作。每个角色运行在独立上下文中，互不干扰。

## 架构说明

每个 `/角色` 命令是一个**调度器**，它会启动一个独立的 Agent 子进程来执行实际工作：
- 子进程拥有独立的 200k 上下文窗口，不受其他角色历史干扰
- 角色间通过 `.claude/tasks/` 目录下的文件进行通信
- 主对话保持轻量，只接收各子进程的摘要

## 可用命令

| 命令 | 角色 | 用途 |
|------|------|------|
| `/项目经理` | 首席架构师 & 技术总监 | 需求分析、**阅读实际代码**、架构设计、任务拆解 |
| `/程序员` | 顶级全栈开发专家 | 代码实现、**可质疑设计缺陷**、自检后交付 |
| `/代码审查员` | 首席质量架构师 | **独立上下文**代码审查、安全审计 |
| `/测试员` | 首席测试工程师 | 测试编写与执行（**可选**，需 conventions.md 定义测试策略） |

## 快速开始

1. **新项目/新任务**: `/项目经理 描述你的需求`（新项目初始化需 main 分支，已有项目在开发分支）
2. **开始开发**: `/程序员 任务目录路径`（开发分支）
3. **代码审查**: `/代码审查员 任务目录路径`（开发分支）
4. **修复审查问题**: `/程序员 任务目录路径`（开发分支，程序员会读取 review-log.md）
5. **测试**（可选）: `/测试员 任务目录路径`（开发分支，需 conventions.md 中定义测试策略）
6. **修复测试问题**: `/程序员 任务目录路径`（开发分支，程序员会读取 test-report.md）
7. **合并到主分支**: 用户手动操作
8. **更新文档**: `/项目经理 更新项目文档`（main/master 分支）

## 目录结构

```
.claude/
├── CLAUDE.md              # 本文件
├── settings.local.json    # 本地权限配置
├── commands/              # 角色命令（SubAgent 调度器）
│   ├── 项目经理.md
│   ├── 程序员.md
│   ├── 代码审查员.md
│   └── 测试员.md
├── templates/             # 文档模板（从 Prompt 中抽离，按需读取）
│   ├── design-template.md
│   ├── dev-tasks-template.md
│   ├── dev-plan-template.md
│   ├── review-log-template.md
│   ├── test-report-template.md
│   └── status-template.md
├── docs/                  # 项目文档（由项目经理维护）
└── tasks/                 # 任务文档目录
    └── {YYYYMMDD_功能名}/
        ├── design.md      # 项目经理写
        ├── dev-tasks.md   # 项目经理写，程序员勾选完成
        ├── status.md      # 项目经理写，程序员/测试员更新进度
        ├── dev-plan.md    # 程序员写
        ├── review-log.md  # 代码审查员写
        └── test-report.md # 测试员写（可选）
```

## 文件权限矩阵

| 文件 | 项目经理 | 程序员 | 代码审查员 | 测试员 | 分支限制 |
|------|:--------:|:------:|:------:|:------:|:--------:|
| docs/*.md | ✅ | ❌ | ❌ | ❌ | 仅 main/master |
| design.md | ✅ | ❌ | ❌ | ❌ | 任何分支 |
| dev-tasks.md | ✅ | ✅（仅勾选 checkbox） | ❌ | ❌ | 任何分支 |
| status.md | ✅ | ✅（仅进度） | ❌ | ✅（仅测试进度） | 任何分支 |
| dev-plan.md | ❌ | ✅ | ❌ | ❌ | 任何分支 |
| review-log.md | ❌ | ❌ | ✅ | ❌ | 任何分支 |
| test-report.md | ❌ | ❌ | ❌ | ✅ | 任何分支 |
| 代码文件 | ❌ | ✅ | ❌ | ❌ | 任何分支 |
| 测试代码文件 | ❌ | ❌ | ❌ | ✅ | 任何分支 |

## 分支规则

- **`tasks/` 目录**：在任何分支上都可以操作（设计、开发、审查都在这里）
- **`docs/` 目录**：**只在 main/master 分支上更新**
  - 原因：docs/ 是项目全局文档，在开发分支上修改会导致分支间冲突或信息不一致
  - 流程：开发分支上完成开发和审查 → 合并到主分支 → 在主分支上执行 `/项目经理 更新项目文档`

## 重要规则

1. **沟通语言**: 所有角色与用户对话使用中文
2. **文件权限**: 各角色只能修改自己负责的文件（见权限矩阵）
3. **项目经理定方向，程序员定实现**: 项目经理负责"做什么"和关键约束，程序员全权决定"怎么做"
4. **项目经理必须读代码**: 设计前必须阅读涉及模块的实际代码，不能只看文档
5. **项目经理必须评估影响**: 涉及现有功能修改时，必须分析影响范围
6. **项目经理必须评估拆分**: 预估改动超过 10-15 个文件时，必须拆分为多个独立任务
7. **程序员可质疑设计**: 发现设计遗漏时必须在 dev-plan.md 中反馈
8. **程序员必须自检**: 代码完成后执行自检清单再交付（含回归检查）
9. **程序员必须更新变更摘要**: dev-plan.md 中的"代码变更摘要"表格供代码审查员定位审查范围
10. **程序员必须勾选任务完成**: 每阶段完成后在 dev-tasks.md 中勾选 checkbox（仅改 `[ ]` → `[x]`）
11. **代码审查员独立审查**: 在隔离上下文中审查，不受开发过程影响
12. **测试员可选参与**: 项目是否需要测试由 conventions.md 中的测试策略决定
13. **测试员只写测试不改业务代码**: 发现 bug 记录到 test-report.md，由程序员修复
14. **设计反馈闭环**: 程序员/代码审查员/测试员反馈设计问题 → 用户决定是否呼叫项目经理修改

## 协作流程

```
用户需求 (开发分支)
    │
    ▼
┌──────────────────┐
│ /项目经理 (子进程)  │
│ 读代码 → 写设计     │    ← tasks/ 目录操作，任何分支都可以
└──────────────────┘
    │ 输出: design.md, dev-tasks.md, status.md
    ▼
┌──────────────────┐     审查回路
│ /程序员 (子进程)  │ ◄───────────────┐
│ 读设计 → 写代码   │                 │
│ 自检 → 交付      │                 │
└──────────────────┘                 │
    │ 输出: dev-plan.md + 代码        │
    ▼                                │
┌──────────────────┐                 │
│ /代码审查员 (子进程)  │                 │
│ 独立审查代码      │                 │
└──────────────────┘                 │
    │                                │
    ├─ 审查有问题 → /程序员修复 ─────┘
    │
    └─ 审查通过
        │
        ▼
┌──────────────────┐ (可选)   测试回路
│ /测试员 (子进程)    │ ◄────────────────┐
│ 写测试 → 跑测试   │                  │
└──────────────────┘                  │
    │                                 │
    ├─ 测试有失败 → /程序员修复 ──────┘
    │               (不经过代码审查员)
    └─ 全部通过
        │
        ▼
┌──────────────────────────────────┐
│ 合并开发分支到 main/master       │  ← 用户手动操作
└──────────────────────────────────┘
        │
        ▼
┌──────────────────┐
│ /项目经理 (子进程)  │    ← 在主分支上执行
│ 更新 docs/ 文档    │    ← docs/ 只在主分支更新
└──────────────────┘
```
